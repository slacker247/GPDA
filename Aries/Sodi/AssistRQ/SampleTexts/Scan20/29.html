Questions

 

Question #1:    What is a NOP slide, and how is this one different from the NOP slide in the rpc.statd exploit in Scan 10?

 

Answer #1:       The NOP slide is a packet with a variety of NOP entries in the payload crafted and transmitted to the target host in an effort to cause a buffer overflow and then injecting the desired payload into active memory in the context of the overrun service user, ideally root for the blackhat.

 

 

Question #2:    The attack was on 08 Jan, 2002.  Would Snort have generated an alert then for the attack?

 

Answer #2:       No.  No Snort attack signature existed for the attack prior to 12 Jan 2002 and CDE is a common service to make available to users across a network.

 

Question #3:    In the exploit code, the command “/bin/sh sh –i” is given, what is its purpose, and why is ‘sh’ shown twice?

 

Answer #3:       This command will cause an interactive shell to be instantiated on a specified port.  Sh is entered twice because this command sequence is usually used when starting a shell on a specified port as root.  This is usually done by making changes to inetd.conf.  So the command uses sh twice, the first time to show what program to pass the second command to.  So /bin/sh is the interpreter, sh is the command –I is the option (interactive).

 

Question #4:    The attacker executed a variety of commands on the hacked Solaris box.  Which commands were automated by the exploit, which commands were manual by the attacker himself?

 

Answer #4:       The exploit itself was automated.  This can be seen in the in the payload of the packet which is very similar to that of the rpc.statd buffer overflow attack.  The same signature was used by the blackhat by starting a remote shell of a target port to simplify their ongoing efforts as they come and go from the server.

 

The commands in the session file for the IP address 208.61.1.160 were most likely either automated or copied and pasted to the connection.  This can be assumed given the length of the first command as well as the failures in some of the commands that follow immediately after; obviously a general script not tailored for this system.  Not to mention that the entire connection lasted 2 minutes and 23 seconds, with a FTP session included in this time, it makes is highly unlikely that this was a manual process.

 

Several unique attempts were made from the compromised host (172.16.1.102) to remote FTP servers.  These were most likely manual given the password failures.

 

The FTP connection to the interland.net FTP appears to have also been automated.  This is supported by the fact that the entire connection takes approximately 15 seconds.  Given that a 90K file was transferred during that time and the commands that were entered by the remote system, supports this being automated.

 

Question #5:    What is sun1, and how does it work?

 

Answer #5:       Sun1 is a tool that the blackhat loads on the compromised machine.  In this instance, it was downloaded from a remote ftp server, moved into /bin/login then chmod’d to 555.  Given its usage, it looks like a Trojan that catches passwords for the blackhat.

 

Question #6:    What did you learn from this exercise?

 

Answer #6:       I never realized how popular the technique of starting a shell on a port was or how easily it was accomplished.  Also, in my research, I found several examples of code.  It also underscored how the tools are getting easier and easier to use and how the skills necessary to be a successful blackhats have lessened.

Bonus:              One of the commands executed during the attack is:

 

Echo “BD PID(s): “`ps –fed|grep’ –s /tmp/x’|grep –v grep|awk ‘{print $2}’`

 

                        What is the purpose of this command and what does ‘BD’ stand for?

 

Answer:            This command is used to identify the PID’s of any process instantiated by the blackhat.  BD is the label the blackhat gave to the processes that he/she started.  Ps –fed lists the active processes.  Grep –s/tmp/x is the string that he/she is searching for in the ps list.  Grep –v grep just causes the columns to be grepped and piped to the awk command.  Grep –v inverts the grep, when followed by an additional grep, it just undoes the –v option; a case of someone being cute.  Finally, the awk statement {print $2} prints the PID of any commands, which meet the grep search criteria.
