Answer to the April 2002 Honeypot challenge

 

By: Chan Chun Fai, Vincent Ip, and Ricci Ieong

 

Analyze of activities

 

After downloading the snort logs from the honeypot challenge webpage, we immediately run the snort log against the md5sum to verify the correctness of the snort log file.

C:\>md5sum 0108@000-snort.log.tar.gz

612be364f54ca5fcb47cf70e69419175 *0108@000-snort.log.tar.gz

 

Afterwards, we used a hex editor to perform the first review of the snort log files. Then we used snort and Ethereal to determine the attacker and capture its commands generated.

 

Identification of the honeypot systems

 

Based on the analysis results, we identified from the snort log results that the range of IP addresses 172.16.1.101 – 172.16.1.109 were being scanned or attacked. Within that range, while 172.16.1.102, 172.16.1.105, and 172.16.1.108 responded to the scanning.

 

Methods used in the investigation

 

In order to answer this April 2002 Honeypot challenge, we tried to test and evaluate the snort logs by simulating the snort environment during the attacked period through three different methods:

§         Running the snort version 1.8.3 which was developed and published in Nov 29, 2001.

§         Running the latest snort version by commenting out the rule that is designed particularly designed the Solaris dtspcd exploit.

§         Running the current snort version but replaced by the ruleset which is published before Jan 08, 2002.

 

snort -vX -r 0108@000-snort.log -c snort.conf -l log > completelog.txt

 

snort -vd -r 0108@000-snort.log -c snort.conf -l log > appslog.txt

 

Based on these three steps of testing, we determined that if snort was run with default setting only, that is without turning on the shellcode rulesets, at the time when the exploit was launched against the honeypot system, no alerts will be generated. Otherwise NOOP shellcode exploits alerts will be generated even without the dtspcd exploits ruleset. Three sets of the alert.ids can be found in the attachment.

Interpretation of the attack

 

According to the timeframe of the attack, it is very likely the attacker used 208.61.69.153 to perform a scan at port 6112 port to the honeypot systems. After he/she identified the target, 172.16.1.102, he/she then performed the EXPLOIT CDE dtspcd exploit at 01/08-22:46:04.378306 - 01/08-22:46:07.042661. Right after he performed 4 consecutive exploit attacks, then the attacker generated the attacks from 208.61.1.160:3596 -> 172.16.1.102:1524. Commands were launched and then he ftp from the ftp server 64.224.118.115 at 01/08-22:46:59.871117 and obtained the backdoor code “sun1” from the server. During that same period, users on 66.156.236.56 also executed a malicious telnet to the honeypot system which in turn generated the telnet alert alarm.

 

Questions

1.      What is a NOP slide, and how is this one different from the NOP slide in the rpc.statd exploit in Scan10?

 

NOP slide is the technique which filling the memory space with NOP followed by exploit code. Doing so is to reduce the effect of different address offset by hoping the address pointer 'lands' on the NOP farm and still can execute the exploit code

 

The NOOP here is SPARC op-code while the NOP used in scan10 is in x86 op-code

 

In addition, in Scan10, the rooms leave for the Assembly code is very small, while in this scan a large buffer is reserved for the attacker's payload, in this case, the attacker has filled up the space with huge amount of 0x41.

As captured by the log, we can see that the complete command that contains “/bin/sh sh –i”:

 

echo “ingreslock stream tcp nowait root /bin/sh sh –I” > /tmp/x

 

 

The purpose of this command is to prepare an entry for inetd to start a service with interactive shell bind. This is because it is used for launching an interactive shell and "/bin/sh sh -i" is to comply with the "inetd.conf" format.  The two 'sh' are used to fulfill the requirement of inetd.conf format.

 

In each inetd.conf entry, the 6th field is the complete process location (in this case '/bin/sh'). And the remaining fields should be the name of the service and it's parameter(s).  In this case, /bin/sh sh –i is being echoed into the temporary file “/tmp/x” which was then executed by command “inetd –s /tmp/x”. (Afterwards, the temporary file was being removed after inetd had been started with the shell binded.)

 

 

d.      The attacker executed a variety of commands on the hacked Solaris box. Which commands were automated by the exploit, which commands were manual by the attacker himself?

 

To answer this question, we traced the session from the attackers 208.61.1.160 (which used the backdoor port ‘ingreslock’) by using Ethereal’s “Follow TCP Stream” function.

 

Commands from session 208.61.1.160:ingreslock

 

 

 

 

From 64.224.118.115 to 172.16.1.102:ftp

 

 

According to our investigation, we use the time between commands executed to differentiate which commands are executed in automatic mode or manually. The commands which executed within a very short period of time, say, 1 second, are very likely to be automatic executed/scripted commands. On the other hand, if the time between commands execution is rather long, say, at least 1 second, it is very likely to be a manually executed commands.

 

Using the above assumption, we view the packet time using the time option “Seconds since previous frame”. We found that the following commands are likely to be executed in automatic mode. This is because these command where used for different purpose but executed within a short period of time. (i.e. they executed within a second).

e.       What is sun1, and how does it work?

 

sun1 is very likely to be a login Trojan. Based on viewing the code directly in the snort logs, some readable codes can be shown in the captured download file.

 

Based on the code review process, we assumed that when the attack comes back to the system, the Trojan program sun1 will check the identity of the attacker. After checking the login identity, it will then transfer the control back to the origin login command located in /usr/lib/libfl.k

 

Besides, we identified that the sun1 exploit code is the exploit code developed and compiled using Solaris elf3 format. Unfortunately, we cannot tested the exploit code because the code was truncated.

 

f.        What did you learn from this exercise?

 

Protective measure is always better than Detective measure! "Protect the unknown" should be a better way in security measure.

 

Advisory is ALWAYS announced after a security hole is found, and the time lap could have given enough time for attacker to write an exploit. Many IDS [like virus scanner] could only help you protect against worldwide known facts.

 

Advanced Attackers can always find a way to exploit network services. Disable unnecessary service is quite crucial. Therefore the best practice is not to run services as "root".

 

           

g.      How long did this challenge take you?

 

Totally 5.0 hours were used to complete this honeypot challenge. 1.5  hours for preliminary investigation, 1 hour for further in-depth investigation, 2.5 hours for write up of the explanations.

 

Bonus Question: One of the commands executed during the attack is

echo "BD PID(s): "`ps -fed|grep ' -s /tmp/x'|grep -v grep|awk '{print $2}'`

 

What is the purpose of this command and what does 'BD' stand for?

 

The purpose of this command is to display the pid of the backdoor program (which is 3476). From the answer in (d), we know that /tmp/x contained the backdoor entry for inetd, and it was started by 'inetd –s /tmp/x'. The command 'ps -fed | -s /tmp/x' gave the process info of inetd that initiated the backdoor.

 

Grep –v grep|awk ‘{print $2}’ filtered the output generated from the 'ps -fed | -s /tmp/x' and obtained the second field which is the PID of the process. Therefore, the command displays the pid of inetd, with the backdoor entry inserted into it.

 

Using this command, the attacker can determine whether the backdoor has been started.

 

BD is probably stand for BackDoor.
